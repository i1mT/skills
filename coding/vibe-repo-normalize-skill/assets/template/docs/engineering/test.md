# 测试规范

测试用于保障项目在持续迭代中的正确性、稳定性和可维护性。所有影响核心逻辑、数据处理、权限控制、用户关键路径的变更，都应补充或更新对应测试。

## 测试类型

- 单元测试：覆盖核心函数、状态转换、数据转换、业务规则和边界条件。
- 集成测试：覆盖模块之间的协作关系，例如 API、数据库访问、服务调用和关键流程。
- 回归测试：针对历史缺陷和高风险路径保留稳定用例，避免重复问题再次出现。

## 执行要求

1. 开发完成后必须运行相关测试，并修复所有失败用例。
2. 涉及公共模块、核心业务或跨模块调用时，应运行完整测试集。
3. 测试失败时，不允许通过跳过测试、删除断言或降低覆盖范围来规避问题。
4. 只有测试全通过后，才可以创建或合并到 `develop` 分支的 PR。

## 测试环境

测试应尽量使用可重复、可隔离的环境。依赖外部服务、数据库或第三方 API 时，应优先使用测试配置、模拟数据或专门的预览环境，避免污染生产数据。
