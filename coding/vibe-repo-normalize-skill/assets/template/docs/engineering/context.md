# GEB 文档管理机制

GEB 用于维护仓库级上下文，让项目在长期迭代中仍然保持可理解、可接手、可追踪。它由三类信息组成：

- `G`：Global，全局项目上下文，记录项目是什么、整体架构、核心约定和重要入口。
- `E`：Entry，目录级入口说明，记录当前目录下每个文件或子目录的职责。
- `B`：Block，代码文件头部说明，记录单个文件在局部系统中的输入、输出和位置。

## 文档类型

1. 主文档：项目根目录的 `AGENTS.md`，用于说明项目级上下文，包括项目目标、技术栈、目录结构、文档系统、开发约定等。
2. 子文档：除项目根目录外，各目录下的 `AGENTS.md`，用于说明该目录内部的文件结构。每条记录保持简短，说明文件名、地位和功能。
3. 文件头注释：常见代码文件顶部的结构化注释，用于说明该文件的 `input`、`output` 和 `pos`。

## 更新原则

1. 任何功能、架构、代码或目录结构变更完成后，必须同步更新受影响目录的子文档。
2. 如果变更影响项目级约定、核心架构、技术栈、启动方式或目录组织，必须同步更新根目录主文档。
3. 如果变更了某个代码文件的职责、依赖、对外输出或在模块中的位置，必须同步更新该文件的头部注释。
4. 文档更新应当跟随代码变更一起完成，不把上下文维护留到之后补做。

## 子文档规范

每个子文档开头必须声明：

```md
一旦我所属的文件夹有所变化，请务必更新我。
```

子文档内容应当覆盖当前目录下的重要文件和子目录。推荐格式：

```md
- `filename.ts`：负责 xxx，是 xxx 模块的入口。
- `components/`：存放 xxx 相关组件。
```

## 文件头注释规范

常见代码文件开头必须包含以下信息：

```ts
// 一旦我被更新，务必同步更新本文件头部注释，以及所属文件夹的 AGENTS.md。
// input: 依赖外部的什么数据、模块、服务或配置。
// output: 对外提供什么能力、组件、函数、接口或副作用。
// pos: 在当前模块或系统局部中的位置与职责。
```

不同语言可按语言习惯调整注释语法，但字段含义必须保持一致。

## 例外规则

1. 对于框架或工具约定不允许出现额外 Markdown 文件的目录，不要创建子文档。
2. 非代码文件、配置文件、自动生成文件、锁文件、静态资源文件不要添加文件头注释。
3. 如果某个目录文件数量很少且职责一目了然，可以只在上级目录的子文档中说明，不必继续下钻生成更多子文档。
