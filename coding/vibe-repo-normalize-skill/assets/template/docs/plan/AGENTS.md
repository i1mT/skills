# 计划文档

本目录用于保存每次迭代前的实现计划。计划文档应帮助团队在动手前明确问题、目标、方案、验证方式和交付动作，避免边做边改导致范围失控。

## 命名规范

计划文件按以下格式命名：

```md
plan/{时间}/{id}.md
```

示例：

```md
plan/2026-06-15/checkout-refactor.md
```

## 计划模板

### 1. 背景

- 要解决什么问题。
- 问题影响哪些用户、角色或业务流程。
- 为什么现在需要处理。

### 2. 当前现状

- 当前功能、代码、数据或流程是什么样子。
- 现有方案有哪些限制、风险或维护成本。
- 已知依赖、历史包袱和不能破坏的行为。

### 3. 目标

- 本次迭代完成后应达到什么状态。
- 明确成功标准，避免只描述“优化”“完善”等模糊目标。
- 明确不在本次处理范围内的内容。

### 4. 实现方案

- 模块拆分：涉及哪些模块、文件夹或服务。
- 架构图：必要时补充系统结构图或数据流图。
- 交互流程：说明模块之间、前后端之间或用户操作之间如何协作。
- 实现步骤：按可提交、可验证的顺序拆分任务。
- API 设计：说明新增或变更的接口、参数、返回值和错误语义。
- 模型设计：说明数据模型、状态模型、权限模型或领域对象变化。
- UI 设计：说明页面结构、关键交互、状态展示和设计系统约束。

### 5. 验证方案

- 测试用例：列出关键正常路径、异常路径和边界条件。
- 测试方式：说明使用单元测试、集成测试、手动验证或端到端验证。
- 验收标准：说明怎样判断实现已经满足目标。

### 6. 交付动作

- 文档：是否需要更新产品、工程、设计或上下文文档。
- 部署：是否需要发布到预览或生产环境。
- PR：是否需要创建 PR、说明风险点和回滚方式。
- 后续事项：记录本次不处理但需要跟进的问题。
